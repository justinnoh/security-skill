# 에러처리 (E-01 ~ E-03)

## E-01 오류 메시지 정보노출 (CWE-209)
- **목적**: 오류 응답으로 내부 정보(스택트레이스/쿼리/경로/버전)가 노출되지 않게.
- **공격 시나리오**: 500 응답에 스택트레이스·SQL·파일경로 노출 → 정찰에 활용.
- **탐지 위치**: 전역 에러 핸들러, `res.send(err)`, `err.stack` 응답, 디버그 모드.
- **PASS**: 사용자에겐 일반화 메시지, 상세는 서버 로그에만 기록, 프로덕션 디버그 off.
- **VULNERABLE**: 예외 객체/스택/내부 메시지를 응답 본문에 그대로 반환.
- **수정**: 표준 에러 응답 포맷, 상세 로깅 분리, `NODE_ENV=production` 처리.

## E-02 오류 상황 대응 부재 (CWE-390/391)
- **목적**: 오류/예외를 감지하고 적절히 처리(무시 금지).
- **공격 시나리오**: 반환값/예외 무시로 인증 실패가 성공처럼 진행.
- **탐지 위치**: 빈 `catch {}`, 무시된 Promise(`await` 누락), 검사 안 된 반환코드.
- **PASS**: 모든 오류 경로 처리, 실패 시 안전한 기본동작(deny).
- **VULNERABLE**: 예외 삼킴(빈 catch), 거부되지 않은 실패.
- **수정**: 명시적 처리, 실패 시 차단, 미처리 거부 핸들러.

## E-03 부적절한 예외 처리 (CWE-248)
- **목적**: 예외를 적절한 단위로 잡고 자원/상태를 일관되게 정리.
- **공격 시나리오**: 과도하게 광범위한 catch로 보안 검증 실패까지 무력화.
- **탐지 위치**: 너무 넓은 `catch(Throwable/Exception)`, 트랜잭션 롤백 누락.
- **PASS**: 구체 예외 처리, finally/리소스 정리, 트랜잭션 롤백.
- **VULNERABLE**: 광범위 포착으로 오류 은폐, 정리 누락.
- **수정**: 구체 예외 캐치, 리소스 정리 보장, 보안 예외는 재전파.
