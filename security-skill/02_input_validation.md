# 입력데이터 검증 및 표현 (I-01 ~ I-17)

각 항목은 다음 필드로 진단한다:
**목적 / 공격 시나리오 / 탐지 위치 / 체크 방법 / PASS 기준 / VULNERABLE 기준 / NEEDS_REVIEW 기준 / 수정 가이드 / 예시**

---

## I-01 SQL 삽입 (CWE-89)
- **목적**: 외부 입력이 SQL 질의 구조를 변경하지 못하게 한다.
- **공격 시나리오**: `' OR '1'='1`, `UNION SELECT`, 스택 쿼리로 인증 우회·데이터 탈취.
- **탐지 위치**: DB 접근 계층, raw query, ORM의 `literal`/`raw`/문자열 결합.
- **체크 방법**: 사용자 입력이 쿼리 문자열에 직접 결합되는지 grep (`SELECT ... +`, 템플릿 리터럴 안 변수).
- **PASS**: 전 쿼리가 파라미터 바인딩/프리페어드 스테이트먼트/ORM 안전 API 사용.
- **VULNERABLE**: 입력을 문자열 결합으로 쿼리에 삽입.
- **NEEDS_REVIEW**: ORM이지만 `raw`/`literal` 사용처가 있어 우회 가능성.
- **수정**: 파라미터 바인딩, 화이트리스트 기반 컬럼/정렬 키, 최소권한 DB 계정.
- **예시(안전)**: `db.query('SELECT * FROM u WHERE id=?', [id])`

## I-02 코드 삽입 (CWE-94)
- **목적**: 입력이 실행 코드로 평가되지 않게 한다.
- **공격 시나리오**: `eval`, `Function()`, `vm`, 템플릿 엔진 SSTI로 임의 코드 실행.
- **탐지 위치**: `eval(`, `new Function(`, `vm.runInContext`, 템플릿 렌더.
- **체크 방법**: 동적 코드 평가 API에 외부 입력이 흐르는지 추적.
- **PASS**: 동적 코드 평가 미사용 또는 입력이 도달하지 않음.
- **VULNERABLE**: 사용자 입력이 `eval`/`Function`/SSTI 경로에 도달.
- **수정**: 동적 평가 제거, 안전한 파서/매핑 테이블, 템플릿 자동 이스케이프.

## I-03 경로 조작 및 자원 삽입 (CWE-22/99)
- **목적**: 입력으로 의도하지 않은 파일/자원 경로 접근 방지.
- **공격 시나리오**: `../../etc/passwd`, 절대경로 주입.
- **탐지 위치**: `fs.readFile`, `path.join(baseDir, userInput)`, 다운로드 핸들러.
- **체크 방법**: 정규화 후 base 디렉토리 이탈 검사 존재 여부.
- **PASS**: `path.resolve` 후 base 접두사 검증 + 화이트리스트.
- **VULNERABLE**: 입력을 경로에 직접 결합, `..` 미검증.
- **수정**: 정규화→base 포함 검증, 식별자→경로 매핑 테이블.

## I-04 크로스사이트 스크립트 / XSS (CWE-79)
- **목적**: 입력이 브라우저에서 스크립트로 실행되지 않게.
- **공격 시나리오**: `<script>`, `onerror=`, DOM sink 주입으로 세션 탈취.
- **탐지 위치**: 서버 템플릿 출력, `innerHTML`, `dangerouslySetInnerHTML`, `v-html`.
- **체크 방법**: 미이스케이프 출력 sink에 입력 도달 여부.
- **PASS**: 출력 컨텍스트별 이스케이프/자동 이스케이프 템플릿, CSP 적용.
- **VULNERABLE**: 입력을 미이스케이프로 HTML에 출력하거나 위험 sink 사용.
- **수정**: 컨텍스트 인코딩, `textContent` 사용, DOMPurify, CSP.

## I-05 운영체제 명령어 삽입 (CWE-78)
- **목적**: 입력이 OS 명령으로 실행되지 않게.
- **공격 시나리오**: `; rm -rf`, `$(...)`, 백틱 주입.
- **탐지 위치**: `exec`, `execSync`, `spawn(shell:true)`, `system`.
- **PASS**: 셸 미경유 `execFile`/배열 인자, 입력은 인자로만 전달.
- **VULNERABLE**: 입력을 셸 문자열에 결합 실행.
- **수정**: 셸 비활성화, 인자 배열 전달, 화이트리스트.

## I-06 위험한 형식 파일 업로드 (CWE-434)
- **목적**: 실행 가능/위험 파일 업로드·실행 차단.
- **공격 시나리오**: `.jsp/.php/.exe` 업로드 후 실행, 이중확장자.
- **탐지 위치**: 업로드 핸들러, multer/formidable 설정, 저장 경로.
- **체크 방법**: 확장자/MIME 화이트리스트, 크기 제한, 저장소 실행권한.
- **PASS**: 화이트리스트 확장자+매직넘버 검증, 비실행 저장소(또는 S3), 랜덤 파일명, 크기 제한.
- **VULNERABLE**: 확장자 검증 없음 또는 블랙리스트만, 웹루트 저장.
- **NEEDS_REVIEW**: MIME만 검사(위조 가능)하고 매직넘버 미검증.
- **수정**: 화이트리스트+콘텐츠 검증, 외부 스토리지, 다운로드시 `Content-Disposition`.

## I-07 신뢰되지 않는 URL 주소로 자동접속 연결 (CWE-601, Open Redirect)
- **목적**: 입력 기반 리다이렉트가 외부 악성 사이트로 가지 않게. (SSRF와 별개 — 클라이언트 리다이렉트)
- **공격 시나리오**: `?next=//evil.com` 피싱 유도.
- **탐지 위치**: `res.redirect(req.query...)`, `location.href = param`.
- **PASS**: 리다이렉트 대상 화이트리스트/상대경로만 허용.
- **VULNERABLE**: 입력을 검증 없이 리다이렉트 대상으로 사용.
- **수정**: 허용목록, 상대경로 강제, 외부 URL은 경유 안내 페이지.

## I-08 부적절한 XML 외부 개체 참조 / XXE (CWE-611)
- **목적**: XML 파서가 외부 개체를 해석해 파일 노출/SSRF되지 않게.
- **탐지 위치**: XML 파서 초기화(`libxml`, `DocumentBuilderFactory`, `lxml`).
- **PASS**: DTD/외부 개체 비활성화.
- **VULNERABLE**: 외부 개체/DTD 허용 기본설정 그대로 사용.
- **수정**: `noent=false`, `disallow-doctype-decl=true`, 외부 엔티티 비활성.

## I-09 XML 삽입 (CWE-91)
- **목적**: 입력이 XML 구조(노드/속성)를 변경하지 못하게.
- **탐지 위치**: 문자열 결합으로 XML 생성, XPath 결합.
- **PASS**: XML 빌더 API + 입력 이스케이프, XPath 파라미터화.
- **VULNERABLE**: 입력을 XML/XPath 문자열에 직접 결합.
- **수정**: 안전한 XML 라이브러리, 특수문자 이스케이프.

## I-10 LDAP 삽입 (CWE-90)
- **목적**: 입력이 LDAP DN/필터를 변조하지 못하게.
- **탐지 위치**: LDAP 검색 필터/DN 문자열 결합.
- **PASS**: DN/필터 이스케이프(RFC 4514/4515), 바인딩 API.
- **VULNERABLE**: 입력을 필터에 직접 결합(`(uid=`+input+`)`).
- **수정**: 특수문자 이스케이프, 화이트리스트.

## I-11 크로스사이트 요청 위조 / CSRF (CWE-352)
- **목적**: 인증 사용자의 의도하지 않은 상태변경 요청 차단.
- **탐지 위치**: 상태변경 라우트(POST/PUT/DELETE)의 토큰/SameSite 적용.
- **PASS**: CSRF 토큰 또는 SameSite=Lax/Strict + 출처 검증.
- **VULNERABLE**: 쿠키 인증인데 CSRF 방어 없음.
- **NEEDS_REVIEW**: 순수 Bearer 토큰(쿠키 미사용)이면 위험 낮음 — 확인 필요.
- **수정**: CSRF 토큰, SameSite 쿠키, Origin/Referer 검증.

## I-12 서버사이드 요청 위조 / SSRF (CWE-918)
- **목적**: 서버가 입력 URL로 내부망/메타데이터에 요청하지 않게.
- **공격 시나리오**: `http://169.254.169.254`, `http://localhost:6379`.
- **탐지 위치**: 서버측 `fetch/axios/http.get(userUrl)`, 웹훅/썸네일/이미지 프록시.
- **PASS**: 대상 호스트 화이트리스트, 사설/링크로컬 IP 차단, 리다이렉트 제한.
- **VULNERABLE**: 입력 URL로 서버가 검증 없이 요청.
- **수정**: 허용목록, DNS 재바인딩 방어, 사설망 차단.

## I-13 HTTP 응답분할 (CWE-113)
- **목적**: 입력의 CR/LF가 응답 헤더를 분할/주입하지 못하게.
- **탐지 위치**: 입력을 헤더/쿠키/리다이렉트 Location에 설정하는 코드.
- **PASS**: 헤더 값 CRLF 제거/거부, 프레임워크 기본 보호.
- **VULNERABLE**: 입력을 헤더에 직접 설정(개행 미제거).
- **수정**: 개행 제거, 안전한 헤더 API.

## I-14 정수형 오버플로우 (CWE-190)
- **목적**: 산술 연산 오버플로우로 인한 버퍼/로직 오류 방지.
- **탐지 위치**: 크기 계산, 인덱스/길이 산술, 금액 계산.
- **PASS**: 범위 검증, 안전 정수 타입/BigInt, 곱셈 전 한계 검사.
- **VULNERABLE**: 외부 입력 기반 크기/길이 산술에 경계 검사 없음.
- **NEEDS_REVIEW**: JS Number 한계(2^53) 초과 가능 금액/카운터.
- **수정**: 입력 상한, BigInt, 체크드 연산.

## I-15 보안기능 결정에 사용되는 부적절한 입력값 (CWE-807)
- **목적**: 인증/인가/가격 등 보안 결정을 신뢰 불가 입력에 의존하지 않게.
- **공격 시나리오**: 클라이언트가 보낸 `role=admin`, `price`, `isVerified` 신뢰.
- **탐지 위치**: 권한/가격/상태를 요청 바디·헤더·쿠키에서 직접 읽는 코드.
- **PASS**: 보안 결정은 서버 세션/DB 권위 데이터로만 수행.
- **VULNERABLE**: 클라이언트 제공값으로 권한/금액/상태 결정.
- **수정**: 서버측 재조회·재계산, 신뢰 경계 명시.

## I-16 메모리 버퍼 오버플로우 (CWE-119/120)
- **목적**: 버퍼 경계를 벗어난 읽기/쓰기 방지(C/C++/네이티브).
- **탐지 위치**: `strcpy/memcpy/sprintf/gets`, FFI/native addon, `Buffer.allocUnsafe`.
- **PASS**: 경계검사·안전 함수, 메모리 안전 언어. (순수 JS/TS면 보통 N/A — 사유 명시)
- **VULNERABLE**: 길이 미검증 복사, `allocUnsafe` 후 미초기화 노출.
- **수정**: 안전 함수(`strncpy`/`snprintf`), 경계검사, `Buffer.alloc`.

## I-17 포맷 스트링 삽입 (CWE-134)
- **목적**: 입력이 포맷 문자열로 해석되지 않게.
- **탐지 위치**: `printf(userInput)`, 로깅 포맷에 입력 직접 전달.
- **PASS**: 포맷 문자열은 상수, 입력은 인자로만 전달.
- **VULNERABLE**: 입력을 포맷 문자열 자리에 전달.
- **수정**: `printf("%s", input)`, 구조적 로깅.
